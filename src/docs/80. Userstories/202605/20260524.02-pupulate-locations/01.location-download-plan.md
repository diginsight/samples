## 2. Schema design — options & decision

### 2.0 Hard requirements

| # | Requirement | Source |
|---|---|---|
| R1 | One Cosmos container only: `location.location` | User request |
| R2 | Each document **must** carry a `partitionKey` property | Container PK = `/partitionKey` |
| R3 | Idempotent re-seed (re-running scripts upserts in place) | Operational |
| R4 | Cover countries world‑wide + admin regions and cities for Europe, the Middle East, North America, India, South America and Oceania + sample addresses | User request |

Everything else (property casing, hierarchy, ID style, geo encoding) is
open for design.

### 2.1 Option matrix

Four structural shapes were considered:

| Option | Shape | Pros | Cons |
|---|---|---|---|
| **A. Fully embedded** | One mega‑doc per country with `regions[]` and `cities[]` embedded | Fewest docs (≈250); a single read returns the full tree | Doc size limit (2 MB) is tight for big countries; updating one city rewrites the whole country; nested arrays are awkward to query/index |
...partition** (**chosen**) | Each entity is its own doc; `partitionKey = <ISO‑2 country code>` for *every* type (country, region, city, address); reference docs (e.g. continent) use `partitionKey = "world"` | "Give me everything about Italy" = single‑partition query; ~250 well‑distributed logical partitions (no hot partition); transactional batches per country are possible; small entity docs are easy to index incl. spatial | Mixing types requires a disciplined `type` discriminator; "all cities in Europe + Middle East + North America + India + South America + Oceania" is cross‑partition (acceptable at the current scale) |
| **C. Type as partition key** | `partitionKey = "country"|"region"|"city"|"address"` | Queries by type are single‑partition | Hot partition for `city` (~5 k docs, all writes hit one LP); "Italy + its cities" needs a cross‑partition query or a secondary filter; doesn't scale if data grows |
| **D. Synthetic composite PK** | e.g. `"city:IT"`, `"country:world"` | Even more granular | No functional benefit over (B); harder to express in queries; breaks one‑shot "Italy" reads |

**Decision: Option B**.

### 2.2 Naming, IDs and geo — secondary decisions

| Topic | Options | Choice | Rationale |
|---|---|---|---|
| Property casing | PascalCase (legacy C# friendly) vs camelCase | **camelCase** | Industry standard for JSON; matches GeoNames REST, REST Countries, Cosmos SDK defaults; C# can opt in with `JsonNamingPolicy.CamelCase` |
| `id` strategy | Random GUID, deterministic GUID v5, **type‑prefixed natural key** | **Type‑prefixed natural key** (e.g. `country-IT`, `region-IT-25`, `city-3173435`, `address-<uuid>`) | Human‑readable, point‑reads in code are trivial (`id`+`pk`), guarantees uniqueness across types in same partition, deterministic ⇒ idempotent |
| Geo encoding | `latitude`/`longitude` floats vs **GeoJSON Point** | **Both**: `location` (GeoJSON Point) is primary; keep `latitude`/`longitude` as convenience | Cosmos supports a spatial index on GeoJSON points ⇒ enables `ST_DISTANCE`, `ST_WITHIN` queries (e.g. “cities within 50 km of...”) |
| Audit metadata | Carry `LocationBase` (`status`, `createdOn`, `createdBy`, ...) on every doc | **Drop**; record provenance in a small `_seed` block | Reference data; audit fields would inflate every doc and add no value. `_seed.source`, `_seed.loadedAt` keep traceability |
| Discriminator | `Type` / `type` / `kind` | **`type`** (camelCase) | Most common, fewest collisions with reserved words |

### 2.3 Final partition strategy summary

Every document carries a `partitionKey` property valued as follows:

| Document type | `partitionKey` value | # logical partitions | Typical \# docs per LP |
|---|---|---|---|
| `continent` | `"world"` | 1 | 7 |
| `country` | ISO‑2 country code (e.g. `"IT"`) | ~250 | 1 |
| `region` | ISO‑2 country code | ~250 | 1–5 (Europe) |
| `city` | ISO‑2 country code | ~250 | 10–80 (Europe; pop ≥ 15 k) |
| `address` | ISO‑2 country code | ~50 | 10–100 |

*All entities about Italy live in the `IT` logical partition. All entities about France live in the `FR` logical partition. Etc.*

### 2.4 Common envelope

```jsonc
{
  "id":           "<type>-<naturalKey>",   // e.g. "city-3173435"
  "type":         "continent|country|region|city|address",
  "partitionKey": "IT|FR|...|world",

  "name":         "Milan",                 // primary display name

  // — type‑specific fields here —

  "_seed": {
    "source":   "geonames:cities15000|mledoze/countries|generated",
    "loadedAt": "2026-05-24T00:00:00Z",
    "version":  "1"
  }
}
```

### 2.10 Indexing recommendation

`ST_DISTANCE(c.location, @point)` queries are cheap on cities and addresses:

```jsonc

  "indexingMode": "consistent",
  "includedPaths": [ { "path": "/*" } ],
  "spatialIndexes": [
    { "path": "/location/?", "types": [ "Point" ] },
    { "path": "/capitalLocation/?", "types": [ "Point" ] }

}
```

## 5. Action plan (step by step)

### Step 0 — Verify the container partition key

```powershell
# Real values for $sub and $account live in samples.internal/.../config/_common.local.ps1
az account set --subscription $sub
$rg = az cosmosdb list `
        --query "[?name=='$account'].resourceGroup | [0]" -o tsv
az cosmosdb sql container show `
    --account-name $account `
    --database-name location `
    --name         location `
    --resource-group $rg `
    --query "resource.partitionKey" -o json
```

- **Expected:** `{ "paths": [ "/partitionKey" ], "kind": "Hash" }`
- **If different:** update the property name used by all `build-*` scripts.

### Step 1 — Get the connection string (kept out of source)

```powershell
$rg   = az cosmosdb list `
            --query "[?name=='$account'].resourceGroup | [0]" -o tsv
$conn = az cosmosdb keys list `
            --type connection-strings `
            --name $account `
            --resource-group $rg `
            --query "connectionStrings[0].connectionString" -o tsv
$env:COSMOS_CONN = $conn   # used by the upload scripts; never committed
```

> The connection string lives **only** in the shell session.

### Step 2 — Download raw datasets (`01-download.ps1`)

- Create `raw/` and `data/` if missing.
- `Invoke-WebRequest` the four URLs into `raw/`.
- `Expand-Archive raw/cities15000.zip data/`.
- Copy plain‑text files (`countries.json`, `countryInfo.txt`,
  `admin1CodesASCII.txt`) directly into `data/`.

### Step 3 — Build `continent` docs (`03-build-continents.ps1`)

- Emit a fixed list of 7 docs (AF / AN / AS / EU / NA / OC / SA).
- Envelope: `id = "continent-<code>"`, `type = "continent"`,
  `partitionKey = "world"`.
- Output → `out/continents.json`.

### Step 4 — Build `country` docs (`04-build-countries.ps1`)

- Read `data/countries.json` (mledoze).
- Project to schema in §2.6 with the common envelope.
- `id = "country-" + cca2`, `partitionKey = cca2`.
- Derive `continentCode` from `region` (mapping table in script).
- Encode `location` and `capitalLocation` as **GeoJSON Point**
  `{ "type":"Point", "coordinates":[lon,lat] }`.
- Output → `out/countries.json`.

### Step 5 — Build `region` docs (`05-build-regions.ps1`)

- Read `data/admin1CodesASCII.txt` (TSV: `CC.RC \t name \t asciiName \t geonameId`).
- Filter to the target country list (Europe + Middle East; see `_common.ps1`).
- Project to §2.7, envelope with `partitionKey = countryCode`.
- `id = "region-{cc}-{adm1}"`.
- Output → `out/regions.json`.

### Step 6 — Build `city` docs (`06-build-cities.ps1`)

- Stream `data/cities15000.txt` (19‑column TSV — see GeoNames readme).
- Filter rows where col 8 (`country code`) ∈ target set (Europe + Middle East).
- Project to §2.8; envelope with `partitionKey = countryCode`.
- `id = "city-{geoNameId}"`.
- Encode `location` as GeoJSON Point.
- Resolve `regionName` from regions built in Step 5.
- Output → `out/cities.json` (≈ 5 000 docs).

### Step 7 — Upload to Cosmos (`99-upload.ps1`)

For each output file, in order (continents → countries → regions → cities):

```powershell
dotnet run --project "..\CosmosdbConsole\CosmosdbConsole.csproj" -- uploadjson `
    -f "out\countries.json" `
    -c $env:COSMOS_CONN `
    -d "location" `
    -t "location" `
    -s "_etag,_rid,_self,_ts,_attachments"
```

Repeat for `regions.json`, then `cities.json`.

### Step 8 — (Optional) Build & upload `address` docs (`07-build-addresses.ps1`)

- Pick N random cities (default 2 000) with a fixed PRNG seed.
- Generate synthetic street/number/postal code.
- Jitter `latitude`/`longitude` by ± ~0.01° and emit matching `location` Point.
- Envelope with `partitionKey = countryCode`, `id = "address-{deterministic-uuid}"`.
- Upload via the same `uploadjson` command.

### Step 9 — Verify

```powershell

  "SELECT VALUE COUNT(1) FROM c WHERE c.type = 'continent'",
  "SELECT VALUE COUNT(1) FROM c WHERE c.type = 'country'",

  "SELECT VALUE COUNT(1) FROM c WHERE c.type = 'city'",
  "SELECT VALUE COUNT(1) FROM c WHERE c.type = 'address'"

# Run via Data Explorer or `az cosmosdb sql query` (preview) /

```

Sanity checks:

- Counts roughly match: continents=7, countries≈250, regions≈1000,

- Random Italian city has `regionName` populated and `location` is a
  GeoJSON Point.
- Spot a country (e.g. `country-IT`): `borders`, `currencies`, `languages`,

## 6. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Container PK differs from `/partitionKey` | Step 0 verifies before any write — confirmed ✅ |
| Mixing types in one container leads to ad‑hoc queries | Every doc carries a `type` discriminator; standard query shape is `WHERE c.partitionKey=@cc [AND c.type=@type]` |
| Cross‑partition queries (e.g. "all countries") | Acceptable at ~250 docs (≪ 50 RU) |
| Re‑running upload duplicates docs | `id`s are deterministic, type‑prefixed natural keys; `UpsertItem` is idempotent |
| Connection string leakage | Never written to disk; passed via `$env:COSMOS_CONN`; `seed-locations/` is gitignored except for scripts |
| Existing C# `Country` / `LocationController` use PascalCase + a `Type` field | Document‑only impact: clients can adopt `JsonNamingPolicy.CamelCase` and update the discriminator field name (`type`) in a follow‑up PR |
| GeoJSON Point requires `[lon, lat]` ordering | Build scripts assemble coordinates explicitly in that order |
