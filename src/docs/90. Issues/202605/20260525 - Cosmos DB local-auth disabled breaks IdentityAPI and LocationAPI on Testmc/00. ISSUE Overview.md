## рЯУЭ DESCRIPTION

While creating a new `Testmc` environment overlay for the Cosmos DB samples (`IdentityAPI`, `LocationAPI`), two blocking problems were uncovered:

1. The target Cosmos DB account `samples-testmc-cdb-itn-01` has **local (key-based) authentication disabled**. The existing controller code instantiated `CosmosClient` directly from a connection string вАФ which requires `AccountKey` вАФ and therefore could not work against this account.
2. The Key Vault `samples-testmc-kv-itn-01` (which is meant to supply the Cosmos connection string at runtime) has **`publicNetworkAccess: Disabled`**. The vault's `networkAcls` (`defaultAction: Allow`, `ipRules: [87.14.32.0/24]`, `bypass: AzureServices`) are ignored while public access is disabled, blocking secret writes from a developer workstation.

### Error Message

Two distinct failure modes were expected/observed:

```
# Runtime вАФ when CosmosClient.GetContainer(...) is called against an AAD-only account using a key-mode client
Microsoft.Azure.Documents.DocumentClientException:
  Request blocked. Local authentication is disabled for this account.
  Use Microsoft Entra ID (Azure AD) authentication instead.
```

```text
# Azure CLI вАФ when attempting az keyvault secret set from a workstation
ERROR: (Forbidden) Public network access is disabled and request is not from a trusted service nor via an approved private link.
Vault: samples-testmc-kv-itn-01; location=italynorth
Inner error: { "code": "ForbiddenByConnection" }
```

### Impact

- All HTTP endpoints on `IdentityAPI` and `LocationAPI` that touch Cosmos (`/User/users`, `/Location/locations/{type}`, `/Location/countries`, etc.) would have thrown immediately on first request against `Testmc`.
- The `appsettings.Testmc.json` files could not be wired to Key Vault (no usable secret existed) вЖТ controllers would have received `""` as `ConnectionString` and thrown `ArgumentException` from the SDK.
- Without a code fix, the only paths forward would have been **(a)** re-enabling Cosmos local auth (regression in security posture) or **(b)** provisioning a separate test account just for samples.

## рЯФђ ANALYSIS

### Root Cause Analysis

#### Primary Cause вАФ code assumes key-based auth

Both controllers were written with the implicit assumption that `CosmosDbOptions.ConnectionString` always contains a full `AccountEndpoint=...;AccountKey=...;` string:

```csharp
this.identityCosmosClient =
    new CosmosClient(identityCosmosDBOptions.ConnectionString);
```

This works against accounts that allow key-based auth, but fails closed against accounts where **local auth is disabled** (Microsoft Entra ID is mandatory). Since the public `samples` repository must remain usable against both classes of accounts (e.g., a developer's personal subscription vs. the shared `samples-testmc-*` infrastructure), the construction logic must support both forms.

#### Secondary Cause вАФ Key Vault is locked down by default

`samples-testmc-kv-itn-01` has `publicNetworkAccess: Disabled`. When this flag is `Disabled`, the more permissive `networkAcls.defaultAction: Allow` and IP rules are inert вАФ the vault only accepts traffic over Private Endpoints or from trusted Azure services (`bypass: AzureServices`). This is the desired production posture, but it blocks the day-to-day "give the developer a way to put a secret in the vault" workflow.

#### Why this only manifests on Testmc

The pre-existing `Testms` environment (`dev-dgw-003-kv`, `cosmosdb-вА¶-eus-001`) had **local auth enabled** and the connection-string secret in Key Vault contained the `AccountKey`. The same code path therefore happened to work вАФ masking the design flaw. `Testmc` is the first environment where Cosmos local auth has been disabled as part of the secure-by-default rollout.

## вЬЕ SOLUTION IMPLEMENTED

### Fix Overview

Two-part fix:

1. **Code** вАФ introduce a private static `CreateCosmosClient(connectionString, logger)` helper in each controller (mirroring the proven implementation already shipping in `CosmosdbConsole/Executor.cs`). The helper inspects the connection string and:
   - Uses `DefaultAzureCredential` when no `AccountKey` is present (either a bare `https://вА¶` endpoint URL **or** an `AccountEndpoint=вА¶;` connection string).
   - Falls back to the classic key-based `new CosmosClient(connectionString)` otherwise.
2. **Secrets** вАФ populate the testmc Key Vault with the **key-less** endpoint connection string. Because the vault has `publicNetworkAccess: Disabled`, this required briefly flipping the vault to `Enabled` (the underlying `networkAcls` retained their original values throughout), performing the writes, then restoring `Disabled`.

### Code Changes

#### 1. `IdentityAPI` вАФ `UserController.cs`

**Location:** [src/03.01 CosmosDB/IdentityAPI/Controllers/UserController.cs](../../../../03.01%20CosmosDB/IdentityAPI/Controllers/UserController.cs)

```csharp
// BEFORE (line 31):

// AFTER:
```

Helper added (private to the controller):

```csharp
/// <summary>
/// Create a CosmosClient supporting both classic AccountKey connection strings
/// and AAD-only endpoints. The <paramref name="connectionString"/> may be:
///   - a full "AccountEndpoint=...;AccountKey=...;" connection string, OR
///   - just a https:// endpoint URL (then DefaultAzureCredential is used), OR
///   - "AccountEndpoint=...;" without AccountKey (then DefaultAzureCredential is used).
/// </summary>
private static CosmosClient CreateCosmosClient(string connectionString, ILogger logger)
{
    var parts = connectionString
        .Split(';', StringSplitOptions.RemoveEmptyEntries)
        .Select(static x => x.Split('=', 2))
        .Where(static x => x.Length == 2)
        .ToDictionary(static x => x[0].Trim(), static x => x[1].Trim(),
                      StringComparer.OrdinalIgnoreCase);

    string? accountEndpoint = null;
    if (parts.TryGetValue("AccountEndpoint", out var ep))
    {
        accountEndpoint = ep;
    }
    else if (Uri.IsWellFormedUriString(connectionString.Trim(), UriKind.Absolute))
    {
        accountEndpoint = connectionString.Trim();
    }

    bool hasKey = parts.ContainsKey("AccountKey");

    if (!hasKey)
    {
        if (string.IsNullOrWhiteSpace(accountEndpoint))
        {
            throw new ArgumentException(
                "Connection string has no AccountEndpoint and is not a bare endpoint URL.",
                nameof(connectionString));
        }
        logger.LogInformation("Using AAD auth (DefaultAzureCredential) for endpoint {endpoint}",
                              accountEndpoint);
        return new CosmosClient(accountEndpoint, new DefaultAzureCredential());
    }

    logger.LogDebug("Using AccountKey auth for endpoint {endpoint}", accountEndpoint);
    return new CosmosClient(connectionString);
}
```

### Solution Features

#### вЬЕ Hybrid auth вАФ single code path, both deployment models

- Pre-existing `Testms` (account-key) deployments continue to work unchanged: the helper detects `AccountKey` and uses the classic constructor.
- New `Testmc` (AAD-only) deployments work transparently: the helper detects the missing key and uses `DefaultAzureCredential`.
- Mirrors the proven [src/03.01 CosmosDB/CosmosdbConsole/Executor.cs](../../../../03.01%20CosmosDB/CosmosdbConsole/Executor.cs) implementation (lines 33вАУ65), so test seed-locations tooling and the runtime APIs converge on a single pattern.

#### вЬЕ Secret wiring stays out of source code

- Secret name follows the standard ASP.NET Core Key Vault provider convention (`--` вЖТ `:`):
  `IdentityApi--CosmosDb--ConnectionString` вЖТ `IdentityApi:CosmosDb:ConnectionString`.
- `appsettings.Testmc.json` retains `"ConnectionString": ""` with a `// Key Vault` comment; the value is supplied at runtime by `ConfigureAppConfiguration2`.

#### вЬЕ Defense-in-depth posture preserved

- Cosmos local auth remains **disabled** in `Testmc`.
- Key Vault `publicNetworkAccess` was returned to `Disabled` immediately after the secret writes; `networkAcls` were never modified.

## рЯУЪ ADDITIONAL INFORMATION

### Required Data-plane Role Assignment

`DefaultAzureCredential` resolves to the developer's signed-in identity (locally) or the App Service managed identity (in `Testmc`). Either way, the identity must be granted the **Cosmos DB Built-in Data Contributor** role at the Cosmos DB account scope:

```powershell
# Role-definition GUID is constant: 00000000-0000-0000-0000-000000000002 (Data Contributor)
az cosmosdb sql role assignment create `
    --account-name samples-testmc-cdb-itn-01 `
    --resource-group samples-testmc-rg-itn-01 `
    --scope "/" `
    --principal-id <objectId-of-user-or-MI> `
    --role-definition-id 00000000-0000-0000-0000-000000000002
```

This is **not** an ARM RBAC role вАФ Cosmos DB has a separate data-plane RBAC system. Granting `Owner` at the management plane does **not** grant data access.

Continue to work вАФ no migration required. The helper detects `AccountKey` and uses the classic path.

### Security Considerations

- вЬЕ **No keys in source code** вАФ both files only contain a key-_handling_ helper; no secret material is checked in.
- вЬЕ **Cosmos local auth stayed disabled** in `Testmc`.
- вЬЕ **Key Vault `publicNetworkAccess` restored** to `Disabled` immediately after the writes.
- вЬЕ **No change to `networkAcls`** вАФ the IP allow-list (`87.14.32.0/24`) was never modified; the temporary window relied on the higher-level `publicNetworkAccess` switch only.
- вЪ†пЄП **Temporary public-access window** вАФ during the brief `Enabled` window, the vault was reachable from any IP because `defaultAction: Allow`. The window was bounded to two `az keyvault secret set` calls (~seconds) and was followed by an explicit `Disabled` write that was verified.

## рЯОУ LESSONS LEARNED

### What Went Wrong

1. **Implicit auth assumption** вАФ both controllers assumed `CosmosDbOptions.ConnectionString` always contained an `AccountKey`. There was no compile-time or startup-time signal that this assumption was being violated under `Testmc`.
2. **Three copies of the same logic** вАФ `CosmosdbConsole` already had a correct hybrid helper, but the API projects re-implemented the naive form. The good pattern wasn't shared.
3. **KV lockdown vs. day-to-day write workflow** вАФ the production-grade `publicNetworkAccess: Disabled` posture is correct, but there was no documented "approved" path for a developer to put a secret in. The pragmatic fix (temporary toggle) is intrinsically a privileged operation that should be auditable.

### What Went Right

1. **Reference implementation existed** вАФ `CosmosdbConsole/Executor.cs` already had `CreateCosmosClient`, which made the fix mechanical and low-risk (drop-in copy).
2. **Backward compatibility preserved** вАФ the helper accepts both classic and AAD forms, so no existing deployment regressed.
3. **Posture restored deterministically** вАФ the KV `publicNetworkAccess` flip was scripted end-to-end with an explicit `Disabled` write at the end, and the final state was diff-verified against the original snapshot.
4. **Multi-target build green** вАФ `0 errors` across `net8.0` / `net9.0` / `net10.0` confirmed the change compiles wherever the samples target.

### Improvements for Future

1. **Promote to shared library** вАФ move `CreateCosmosClient` (and its tests) into `Diginsight.Components.Azure`; have all samples depend on the shared helper.
2. **Detect at startup** вАФ add a `Configure<CosmosDbOptions>` post-configure that fails fast (with an actionable message) if `ConnectionString` is empty and no KV is wired.
3. **Codify the KV write workflow** вАФ capture the "temporarily enable public access, write, restore" pattern as a reusable script in `infra/scripts/` rather than ad-hoc PowerShell. Better: add a Private Endpoint reachable from a build agent.
4. **IaC for data-plane RBAC** вАФ represent the Cosmos DB Data Contributor assignment in Bicep so that AAD-only Cosmos accounts are usable from `Day 0` without a manual follow-up step.
5. **Documentation** вАФ add a one-pager under `docs/03. UseCases/` explaining how `Testms` and `Testmc` differ and the implications for sample authors.
