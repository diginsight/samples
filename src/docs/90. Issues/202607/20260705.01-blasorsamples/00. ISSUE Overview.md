## 📝 DESCRIPTION

While bringing up the newly created **02.01_ReactSample** (a React + TypeScript + Vite SPA
that mirrors the existing `01.01_BlazorSample` architecture), the interactive sign-in flow
failed immediately after clicking **Sign in**. Microsoft Entra ID rejected the authorization
request because the redirect URI used by the Vite dev server (`http://localhost:5173`) was
not registered on the SPA app registration `samples-testmc-appreg-01`.

The React SPA obtains its MSAL configuration at startup from the API's anonymous
`/api/clientconfig/auth` endpoint and uses `window.location.origin` as its `redirectUri`.
On the Vite dev server this origin is `http://localhost:5173`, but the app registration only
had the **Blazor** WebAssembly callback URIs registered (`.../authentication/login-callback`).

### Error Message
```
AADSTS50011: The redirect URI 'http://localhost:5173' specified in the request does not
match the redirect URIs configured for the application 'adb2af34-676b-4773-b592-2702de9a5388'.
Make sure the redirect URI sent in the request matches one added to your application in the
Azure portal. Navigate to https://aka.ms/redirectUriMismatchError to learn more about how to
fix this.
```

### Impact
- ❌ Users cannot sign in to the React sample; the app is unusable past the login screen.
- ❌ The protected `GET /weatherforecast` endpoint cannot be exercised (no access token).
- ✅ No impact to the existing `01.01_BlazorSample` — its callback URIs were untouched.

## 🔬 ANALYSIS

### Root Cause Analysis

#### Primary Cause — Missing SPA redirect URI on the app registration
The React sample reuses the **same** SPA app registration as the Blazor sample
(`samples-testmc-appreg-01`). The Blazor WebAssembly client registers redirect URIs of the
form `{origin}/authentication/login-callback` on ports `5049/7259/5241/7123`. The React SPA,
by MSAL convention, redirects back to the **bare origin** `http://localhost:5173` (the Vite
dev server port). Because Entra ID requires an **exact** string match between the
`redirect_uri` in the request and one registered on the application, the authorization
request was rejected with `AADSTS50011`.

#### Why the mismatch occurred
- Blazor WASM MSAL uses a dedicated callback **path** (`/authentication/login-callback`).
- Browser MSAL (`@azure/msal-browser`) used by the React SPA defaults to the page origin as
  the redirect URI (`window.location.origin`), with **no path**.
- The two frameworks therefore need **different** redirect URIs even when sharing one app
  registration, and the React one had never been added.

#### Error Manifestation
```
1. User opens http://localhost:5173 and clicks "Sign in".
2. SPA calls msalInstance.loginRedirect({ scopes: [...] }).
3. MSAL builds an /authorize request with redirect_uri = http://localhost:5173.
4. Entra ID compares it against the app's registered redirect URIs → no match.
5. Entra ID returns AADSTS50011 and refuses to sign the user in.
```

### Impact Assessment

| Category | Impact | Severity |
|----------|--------|----------|
| **Functionality** | Sign-in blocked; SPA cannot authenticate | High (for the sample) |
| **Data Integrity** | None — no data mutated | None |
| **User Experience** | Hard stop at login with a raw AAD error | Medium |
| **Blast radius** | Limited to the new React sample in dev | Low |

### Affected Workflows
1. ❌ **Interactive sign-in (React SPA)**: `loginRedirect` rejected by Entra ID.
2. ❌ **Access token acquisition**: no token → protected API calls impossible.
3. ✅ **Blazor sample sign-in**: unaffected; its callback URIs remained valid.

## SOLUTION ANALYSIS

### Fix Overview
Add `http://localhost:5173` as a **Single-page application** redirect URI to
`samples-testmc-appreg-01`, preserving the four existing Blazor callback URIs. The change was
applied via Microsoft Graph (additive, reversible). No code change was required for the fix,
though the requirement is now documented in the sample README.

### Code / Configuration Changes

#### 1. Add the React SPA origin to the app registration (Microsoft Graph PATCH)
**Target:** `applications/{objectId}` where `objectId = 0a7d79f2-d1da-4557-bba6-4cf5cfdb0631`

```jsonc
// PATCH body — full list (four existing + one new)
{
  "spa": {
    "redirectUris": [
      "http://localhost:5049/authentication/login-callback",
      "https://localhost:7259/authentication/login-callback",
      "http://localhost:5241/authentication/login-callback",
      "https://localhost:7123/authentication/login-callback",
      "http://localhost:5173"
    ]
  }
}
```

```powershell
# Applied with Azure CLI (signed in as tenant admin)
az rest --method PATCH `
  --uri "https://graph.microsoft.com/v1.0/applications/0a7d79f2-d1da-4557-bba6-4cf5cfdb0631" `
  --body "@spa-redirect-uris.json"
```

#### 2. Documented the requirement in the sample README
**Location:** `src/30.01 Web/02.01_ReactSample/README.md`

```markdown
### Entra redirect URI
The SPA app registration (`samples-testmc-appreg-01`) must have `http://localhost:5173`
registered as a **Single-page application** redirect URI. Add it under
*App registrations → samples-testmc-appreg-01 → Authentication → Single-page application*
if sign-in fails with `redirect_uri` mismatch. Use `http://localhost:4173` for `npm run preview`.
```

### Verification (after fix)
```
spa.redirectUris =
[
  "http://localhost:5173",                                    ← added
  "http://localhost:5049/authentication/login-callback",
  "https://localhost:7259/authentication/login-callback",
  "http://localhost:5241/authentication/login-callback",
  "https://localhost:7123/authentication/login-callback"
]
```

### Solution Features
#### ✅ Additive and reversible
- Existing Blazor callback URIs preserved; only one URI added. Easily removed if needed.

#### ✅ No secret exposure
- Redirect URIs and client ids are public identifiers, not secrets.

### Testing Recommendations

   redirect returns to the SPA and `AuthenticatedTemplate` renders the Agent page.
2. **Token/API test** — confirm `GET /weatherforecast` returns `200` with a bearer token

3. **Preview build** — run `npm run preview` (port `4173`) and verify sign-in; add
   `http://localhost:4173` to the SPA redirect URIs if that flow is required.

### Migration Considerations
#### ⚠️ Important: per-environment redirect URIs
Each hosted environment (dev/preview/staging/prod) needs its **own** SPA redirect URI
registered. When deploying the React sample, add the deployed origin (e.g.
`https://<app>.azurewebsites.net`) to the app registration.

#### Options to avoid future clashes
- **Option 1 — dedicated app registration** for the React SPA (cleanest separation from Blazor).
- **Option 2 — shared app registration** (current approach) with all framework-specific
  redirect URIs enumerated; document each one.

### Security Considerations
- ✅ **Exact-match redirect URIs**: Entra's strict matching is a security feature preventing
  token redirection to untrusted origins — keep the list minimal and intentional.
- ✅ **`sessionStorage` cache**: MSAL is configured with `cacheLocation: "sessionStorage"`,
  reducing token persistence surface.
- ⚠️ **HTTP on localhost**: acceptable for local dev only (localhost is a secure context);
  production must use HTTPS origins.

## 🎓 LESSONS LEARNED

### What Went Wrong
1. **Framework redirect conventions differ**: reusing the Blazor app registration overlooked
   that browser MSAL redirects to the bare origin while Blazor WASM uses a callback path.
2. **Redirect URI not provisioned up front**: the sample was runnable before the identity
   side was fully configured, surfacing the error only at sign-in.

### What Went Right
1. **Clear diagnostic**: `AADSTS50011` names the exact offending `redirect_uri` and client id.
2. **Additive, reversible fix**: the Graph PATCH preserved existing URIs — zero risk to Blazor.
3. **Documented for next time**: the README now calls out the redirect URI requirement.

### Improvements for Future
1. **Provision identity with the sample**: add required redirect URIs as part of scaffolding a
   new SPA sample.
2. **Prefer a dedicated app registration** per SPA to avoid cross-framework redirect coupling.
3. **Pre-flight checklist**: verify `spa.redirectUris` contains the dev origin before first run.
